# Deployment Guide — AP Automation Platform (Azure Container Apps)

This guide deploys the full stack to Azure:

| Component | Azure service |
|-----------|---------------|
| FastAPI backend (API) | Container App `invoice-backend` (public) |
| Celery worker | Container App `invoice-worker` |
| Celery beat scheduler | Container App `invoice-beat` |
| React/Vite frontend | Container App `invoice-frontend` (public) |
| Database | Azure Database for PostgreSQL Flexible Server |
| Broker / cache | Azure Cache for Redis |
| Uploaded files (shared) | Azure Storage File Share, mounted at `/app/uploads` |
| Image storage | Azure Container Registry (ACR) |

Backend and worker **share** the uploads file share so the worker can read files
the API wrote. Migrations run automatically on backend startup
(`RUN_MIGRATIONS=true`).

## 1. One-command deploy (recommended)

From the **repo root**:

```powershell
# 1. Open infra/deploy.ps1 and edit the variables at the top:
#    - $PGPASS and $SECRET_KEY (use strong values!)
#    - $ACR / $PG / $REDIS / $STG must be GLOBALLY UNIQUE — change suffixes if needed
# 2. Provide your OpenAI key:
$env:OPENAI_API_KEY = "sk-...."
# 3. Run it:
./infra/deploy.ps1
```

The script provisions every resource, builds both images **in ACR** (no local
Docker needed), and deploys all four container apps. Redis + Postgres creation
take ~15-20 min, so expect the run to take a while.

When it finishes it prints the **Frontend** and **Backend** URLs. Then seed the
database once:

```powershell
az containerapp exec -g invoice-p2p-rg -n invoice-backend --command "python seed/seed.py"
```

Default login after seeding: **admin@company.com / password123**

> Skip to [section 4](#4-set-up-cicd-github-actions) for CI/CD, or
> [section 5](#5-going-fully-production-grade-hardening) for hardening.

## 2. Manual step-by-step (Azure CLI)

If you prefer to run each command yourself (or to understand what the script does):

```powershell
# Variables
$LOCATION="eastus"; $RG="invoice-p2p-rg"; $ENVNAME="invoice-p2p-env"
$ACR="invoicep2pacr"; $PG="invoice-p2p-pg-001"; $REDIS="invoice-p2p-redis-001"; $STG="invoicep2pstor001"
$PGADMIN="apadmin"; $PGPASS="Ch4nge-Me-Str0ng!"; $PGDB="ap_platform"
$env:OPENAI_API_KEY="sk-..."

# Providers + extension
az extension add --name containerapp --upgrade
az provider register -n Microsoft.App --wait
az provider register -n Microsoft.OperationalInsights --wait

# Resource group + registry
az group create -n $RG -l $LOCATION
az acr create -n $ACR -g $RG --sku Basic --admin-enabled true

# PostgreSQL
az postgres flexible-server create -g $RG -n $PG -l $LOCATION `
  --admin-user $PGADMIN --admin-password $PGPASS `
  --sku-name Standard_B1ms --tier Burstable --storage-size 32 --version 16 `
  --public-access 0.0.0.0 --yes
az postgres flexible-server db create -g $RG -s $PG -d $PGDB
az postgres flexible-server parameter set -g $RG -s $PG --name require_secure_transport --value OFF

# Redis (~15-20 min)
az redis create -n $REDIS -g $RG -l $LOCATION --sku Basic --vm-size c0 --enable-non-ssl-port

# Storage + file share
az storage account create -n $STG -g $RG -l $LOCATION --sku Standard_LRS
$STG_KEY=(az storage account keys list -n $STG -g $RG --query "[0].value" -o tsv)
az storage share-rm create -g $RG --storage-account $STG --name uploads --quota 50

# Container Apps environment + storage mount
az containerapp env create -n $ENVNAME -g $RG -l $LOCATION
az containerapp env storage set -g $RG -n $ENVNAME --storage-name uploadsmount `
  --azure-file-account-name $STG --azure-file-account-key $STG_KEY `
  --azure-file-share-name uploads --access-mode ReadWrite

# Build images in ACR
$ENV_DOMAIN=(az containerapp env show -n $ENVNAME -g $RG --query properties.defaultDomain -o tsv)
$BACKEND_URL="https://invoice-backend.$ENV_DOMAIN"
az acr build -r $ACR -t invoice-backend:latest -f backend/Dockerfile .
az acr build -r $ACR -t invoice-frontend:latest -f frontend/Dockerfile ./frontend --build-arg VITE_API_URL=$BACKEND_URL
```

Then create the four container apps. The exact YAML (with secrets, the uploads
volume mount, and the worker/beat command overrides) is generated by
`infra/deploy.ps1` — copy that logic, or just run the script. The apps are:

- `invoice-backend` — ingress external, port 8000, `RUN_MIGRATIONS=true`, uploads mount
- `invoice-worker` — no ingress, command `celery ... worker`, uploads mount
- `invoice-beat` — no ingress, command `celery ... beat`, single replica
- `invoice-frontend` — ingress external, port 80

## 3. Manual step-by-step (Azure Portal)

The portal can't mount file shares or set worker commands as cleanly as the CLI,
but here is the click-path. **CLI/script is strongly recommended for the apps.**

1. **Resource group** — Portal → *Resource groups* → *Create* → `invoice-p2p-rg`.
2. **Container Registry** — *Create a resource* → *Container Registry* → Basic
   SKU. After creation, *Access keys* → enable **Admin user**.
3. **PostgreSQL** — *Create* → *Azure Database for PostgreSQL Flexible Server* →
   Burstable B1ms, v16, admin user/password. Under *Networking* enable
   *Allow public access from Azure services*. Create a database `ap_platform`.
   Under *Server parameters* set `require_secure_transport = OFF`.
4. **Azure Cache for Redis** — *Create* → Basic C0. Under *Advanced settings*
   enable the **non-TLS port (6379)**. Copy the **primary access key**.
5. **Storage account** — *Create* → Standard LRS. *File shares* → *Create* a
   share named `uploads`.
6. **Container Apps Environment** — *Create* → *Container Apps Environment*.
   After creation, open it → *Settings → Azure Files* and add the `uploads`
   share (name it `uploadsmount`, ReadWrite).
7. **Build & push images** — easiest from CLI: `az acr build ...` (see section 2).
8. **Container Apps** — *Create* → *Container App* four times. For each, choose
   the environment, the ACR image, set env vars/secrets (DB/Redis/OpenAI), and
   for backend+worker add the `uploadsmount` volume mounted at `/app/uploads`
   (App → *Volumes*, then *Containers → Volume mounts*). Set worker/beat
   *Command override* to the celery commands. Enable ingress (external) for
   backend (8000) and frontend (80) only.

> Because of the volume mounts and command overrides, doing step 8 by hand is
> error-prone. Use `infra/deploy.ps1`.

## 4. Set up CI/CD (GitHub Actions)

The workflow in [.github/workflows/deploy.yml](.github/workflows/deploy.yml)
builds both images in ACR and rolls them onto the existing container apps on
every push to `main`.

Create a deployment service principal and add GitHub secrets:

```powershell
$SUB=(az account show --query id -o tsv)
az ad sp create-for-rbac --name invoice-p2p-ci --role contributor `
  --scopes /subscriptions/$SUB/resourceGroups/invoice-p2p-rg --sdk-auth
# copy the entire JSON output -> GitHub secret AZURE_CREDENTIALS
```

In GitHub → repo **Settings → Secrets and variables → Actions**, add:

| Secret | Value |
|--------|-------|
| `AZURE_CREDENTIALS` | the full JSON from the command above |
| `ACR_NAME` | `invoicep2pacr` |
| `RESOURCE_GROUP` | `invoice-p2p-rg` |
| `BACKEND_URL` | `https://invoice-backend.<region>.azurecontainerapps.io` |

Push to `main` → the workflow builds and deploys automatically.

## 6. Going fully production-grade (hardening)

The defaults above prioritise "works end-to-end". Before real production:

1. **Re-enable TLS:**
   - Postgres: set `require_secure_transport = ON`; append `?ssl=require` to
     `DATABASE_URL` (asyncpg) and configure the sync (pg8000) SSL context.
   - Redis: disable the non-SSL port; use `rediss://...:6380/...` and configure
     Celery `broker_use_ssl`.
2. **Secrets** → move from container-app secrets to **Azure Key Vault** and
   reference via managed identity; rotate `SECRET_KEY`, DB password, OpenAI key.
3. **ACR auth** → use **managed identity** instead of admin username/password.
4. **Database** → bump tier off Burstable, enable HA + automated backups, lock
   networking to the Container Apps subnet (Private Endpoint / VNet integration).
5. **Custom domain + TLS cert** on the frontend container app.
6. **Uploads** → consider migrating from the file-share mount to **Azure Blob
   Storage** (the codebase already has an `AZURE_STORAGE_CONNECTION_STRING`
   setting and a storage-service abstraction to build on).
7. **Observability** → wire Log Analytics / Application Insights; set scaling
   rules (e.g. worker scales on Redis queue length via KEDA).

## 7. Troubleshooting

| Symptom | Fix |
|---------|-----|
| Backend won't start / DB errors | Check `az containerapp logs show -g invoice-p2p-rg -n invoice-backend --follow`. Verify Postgres firewall allows Azure services and `require_secure_transport=OFF`. |
| Worker not processing | Confirm Redis non-SSL port enabled and `CELERY_BROKER_URL` uses `redis://...:6379`. Check worker logs. |
| Frontend loads but API calls fail (CORS) | Ensure backend `ALLOWED_ORIGINS` includes the frontend URL and the frontend image was built with the correct `VITE_API_URL`. |
| Uploaded file "not found" in worker | Both backend and worker must mount the same `uploadsmount` share at `/app/uploads`. |
| Image build fails on OCR libs | The backend Dockerfile installs tesseract/poppler/libgl — rebuild with `az acr build`. |
